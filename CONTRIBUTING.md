# 贡献指南

**感谢你乐意为我们的项目做出贡献!**

在做出贡献前, 你需要**阅读并遵守**该贡献指南, 以保证项目有组织有规范.

---
## 📝 整体规范

***所有文本, 一律写半角标点符号+空格, 不尾随句号.***

## 💡 Issue

### 基本原则
当你有**希望添加或改动的功能**, 以及**希望解决的 bug** 时, 你可以提出 Issue.

**创建任何分支前**, 绝大多数情况下你都**应该**提出一个 Issue, 描述你要做什么.
新建的分支绝大多数都有一个关联的 Issue. **即使分支体量很小** (如只有一行的改动),
你也**应该**提个 Issue.

Issue 是**原子性的**, 多个不相关的请求或报告应该**分开提多个 Issue**.

**提出问题时**, 请**针对主分支**的问题提出 Issue, 而不要针对其它分支, 它们都是短命分支.

### 标题
Issue 的标题应**简洁明确**: 一句话概括核心问题或需求.

你应当给 Issue 的标题**添加前缀**, 说明该 Issue 的类型:
- `[Bug]` 报告代码问题
- `[Feature]` 请求功能, 特性
- `[Docs]` 报告文档相关的问题

示例:
- ✅ `[Feature] 添加章节选择界面`
- ✅ `[Bug] 故事播放界面布局错乱`
- ❌ `新功能请求`
- ❌ `故事播放有问题`

### 内容
Issue 的具体内容依据其类型而定.

#### 报告代码问题
示例 (模板):

```markdown
### 描述
简要描述 bug 的表现

### 复现步骤
1. 启动 '...'
2. 点击 '...'
3. 再点击 '...'
4. 出现错误

### 预期行为
清晰描述期望的结果

### 实际行为
描述实际发生了什么, 可附上错误信息

### 建议方案 (可选)
描述你期望的修复方式

### 截图/日志（可选）
粘贴截图或控制台/日志输出

### 环境信息 (可选)
- 操作系统: Windows 10
```

#### 功能请求
示例 (模板):

```markdown
### 背景
描述当前存在的痛点或使用场景

### 建议方案
描述你期望的功能或实现方式

### 备选方案 (可选)
列出考虑过的其他方案

### 附加信息 (可选)
截图、参考链接或原型设计
```

## ⛓ 分支管理

**如果你有权限直接管理仓库分支, 那么你应当遵守这些约定.**

### 基本原则
在编写新代码时, **每一个不相关的, 原子性的新功能/改动/修复, 都要创建新分支.**
新功能是新功能, 修 bug 是修 bug, **不要将多个功能放入同一个分支内编写.**

无论一个特性分支**只有你独自开发**, 还是**多人协作**, 都应该在此分支的基础上,
创建出自己的分支. 原有的特性分支称为**集成分支**.

***严禁不提 PR 直接推送至集成分支!*** 你只能直接推送到**自己的个人分支**.
合并个人分支到该集成分支时, **必须提 PR**, 附以详细描述, 然后等待审查通过.

当集成分支需要**多人频繁协作**时, 若容易出现代码冲突 (涉及同一文件的多个改动),
则应**尽可能多交流, 频繁提交合并请求**, 最大程度避免合并冲突.

### 分支命名
分支名称一律使用**全小写英文数字+横杠+正斜杠**的命名规则, 通常是形如 `category/name` 的名称.

`category` 用于表明该分支属于哪些类型的开发. 以下是 `category` 的**通常参数**:

| 类型             | 使用场景                | 示例                                   |
|----------------|---------------------|--------------------------------------|
| **`feature`**  | 新功能开发               | `feature/potential-calculator`       |
| **`fix`**      | 修复 bug 或修改资源文件      | `fix/screen-metrics`                 |
| **`hotfix`**   | 更着急的 fix            | `hotfix/res-loc-audio-category`      |
| **`test`**     | 新增或修改单元测试           | `test/difficulty`                    |
| **`docs`**     | 新增或修改文档, 注释或 .md 文件 | `docs/contributing`                  |
| **`chore`**    | 修改其它配置              | `chore/upgrade-jackson`              |
| **`refactor`** | 代码逻辑重构, 但无业务影响      | `refactor/remove-deprecated-classes` |
| **`style`**    | 代码规范修改, 但没有任何逻辑变动   | `style/optimize-imports`             |
| **`perf`**     | 性能优化                | `perf/story-player-animation`        |
| **`archive`**  | 毙掉了, 但是想保留作纪念       | `archive/master`                     |

`name` 用于说明该分支主要内容, **不宜过长**, 可以适当缩写.

若该分支有对应的 Issue (大部分情况都有), **应当添加** `xxx-` 的 Issue 编号前缀,
以关联 Issue, 如 `fix/12-replace-image-placeholders`

对于个人分支, 应在原有集成分支名称的基础上**添加后缀** `-xxx`, 用于标识这是自己开发的分支,
如 `refactor/13-resource-loader-rikka`

### 合并分支

通过提交 PR 来合并个人分支进集成分支, 或者合并集成分支进主分支.

有关规则见 [提交 PR](#-提交-pr).

### 生命周期
主分支 `main` **永久存在**, 遗留分支 `archive/master` **不再使用但仍保留**.

对于某个集成分支, 若其已经通过 PR 合并到主分支, 则应**及时清理**. 对于后续与之相关的一切变动,
应当**另外创建分支**.

对于集成分支附属的个人分支, 通常**跟随集成分支的生命周期**.

## 📰 Git 提交规范

### 基本原则
一次提交应当是**原子性的, 独立的, 能运行的最小单元**. 不应该把多个功能放在一起提交,
也不应该把报错的半成品提交上来.

若需要提交尚未完成的业务逻辑, 首先应**解决所有编译错误**, 例如注释掉报错的地方.

### 提交消息
提交消息应满足如下格式:

```text
[提交类型]: [具体消息]
```

以下是可用的提交类型:

| 类型             | 使用场景                | 示例                               |
|----------------|---------------------|----------------------------------|
| **`feat`**     | 新功能                 | `feat: 完整故事播放功能实现`               |
| **`fix`**      | 修复 bug 或修改资源文件      | `fix: 屏幕长宽机制导致的布局错误`             |
| **`test`**     | 新增或修改单元测试           | `test: 添加 Difficulty 的单元测试`      |
| **`docs`**     | 新增或修改文档, 注释或 .md 文件 | `docs: 修改代码规范关于空行的要求`            |
| **`chore`**    | 修改其它配置              | `chore: 更新 Jackson 版本为 2.21.3`   |
| **`refactor`** | 代码逻辑重构, 但无业务影响      | `refactor: 提取故事节点重复使用的逻辑为公共静态方法` |
| **`style`**    | 代码规范修改, 但没有任何逻辑变动   | `style: 移除多余 import 语句`          |
| **`perf`**     | 性能优化                | `perf: 提升故事播放器的动画性能`             |

具体消息不宜过长, 但要**说明这次提交具体干了什么**, 不要写过于笼统的消息:

❌ 错误写法:
- `feat: Story 类`
- `fix: 资源路径 bug`
- `refactor: 重写故事显示逻辑`
- `perf: 优化动画`

✅ 推荐写法:
- `feat: 表示故事的 Story 数据类`
- `fix: ResourceLocation.audio() 返回的实例中 category 不正确`
- `refactor: 将故事显示解耦为 View 层与 ViewModel 层`
- `perf: 提升故事播放动画的性能`

对于 `feat`, `fix`, `refactor`, `perf`, **不应当**添加前缀 `添加`, `修复`,
`重构`, `优化`, 而是省去动词或是换用更具体的行为动词. 对于开头的动词, 也应使用**祈使语气**,
而不用过去式: `补充 Difficulty 的单元测试` 而不是 ~~`补充了 Difficulty 的单元测试`~~.

## 🔗 提交 PR

### 基本原则
提交 PR 前, 请确保你要合并的功能都**已经写完**, 不要把半成品扔进去.

任何 PR 都必须指定**至少一位审查者**, 并等待审查通过才能通过合并请求.

### 标题
类似 Git 提交消息规范, 把**最主要的改动**写出来, 具体内容放在描述内.
见 [提交消息](#提交消息).

### 描述
一份 PR 的描述必须包含其**所有改动**. 可以使用序号 (1, 2, 3...) 来逐个说明改动.

可以说明申请此 PR 的背景和原因, 如**为什么添加该特性**, **为什么修改此功能**.

### 跟进工作
**提交了 PR 不等于你可以不管了.** 你需要接受审查者的审查, 并根据其提出的建议进行**改动**,
然后提交推送. 直到审查者**同意 (Approved) 通过该 PR**, 你才能结束该分支的工作.

### 合并
**如果你有权限通过合并请求, 那么你应当遵守这些约定.**

只有在确保**审查没有问题**时才能通过合并请求. 你需要检查:
- 改动能否通过编译
- 存不存在 bug
- 提交符不符合规范

若没有问题, 则可以通过合并请求.

合并时, **选择 `Squash and merge`**, 然后指定提交消息与附加消息.

合并的提交消息与 PR 标题**保持一致** (或是略微修改以符合规范), 附加消息将所有改动附上,
并**关闭**该分支对应的 Issue (若有). 随后, 若该分支确认不再使用, 则应当**及时清理**.
