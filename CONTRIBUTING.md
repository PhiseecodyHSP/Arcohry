# 贡献指南

**感谢你乐意为我们的项目做出贡献!**

在做出贡献前, 你需要**阅读并遵守**该贡献指南, 以保证项目有组织有规范.

---
## 📝 整体规范

我们推荐, 写消息文本时:
- 使用**半角标点符号**, 并**尾随空格**
- 中文和英文, 数字之间**用一个空格隔开**

这样可以让文本更加美观, 当然**并不是强制要求**.

## 💡 Issue

### 基本原则
*由于 Issue 可以记录计划功能和改动, 遵循这些原则可以使开发更有效率.*

Issue 是**原子性的**, 多个不相关的请求或报告应该**分开提多个 Issue**,
这样便于讨论.

**提出问题时**, 大多数情况请**针对主分支**的问题提出 Issue. 不建议向其它分支提出 Issue, 
因为它们很多时候是单人开发, 体量也不会很大, 一般会在 PR 中评论以指出问题所在.

提 Issue 请**文明用语**, 辱骂并不能促使开发者接受你的请求.

### 什么时候提 Issue
当你有**希望添加或改动的功能**, 以及**希望解决的 bug** 时, 你可以提出 Issue.

**创建分支前**, 大多数情况下, 例如涉及业务逻辑改动时, 都会提出一个 Issue, 描述你要做什么,
然后与分支关联.

体量小的润色改动**不必**提 Issue, 尤其是**非业务改动**, 因为 "你帮别人补了个空格" 这种事不必在意.

例如:
- 增加新功能, **应该**提一个 Issue
- 修复项目 bug, **应该**提一个 Issue
- 修改几个业务参数, **应该**提一个 Issue
- 修正几个文件的代码规范, **不需要**提一个 Issue

这么做的好处是:
- 在开工前能充分获取意见, 探究改动的合理性
- 每次业务改动都有 Issue 追踪

### 标题
Issue 的标题要**简洁明确**: 一句话概括核心问题或需求.

推荐给 Issue 的标题**添加前缀**, 说明该 Issue 的类型, 例如:
- `[Bug]` 报告代码问题
- `[Feature]` 请求功能, 特性
- `[Docs]` 报告文档相关的问题

这样方便开发者了解不同 Issue 的类型和主要内容.

示例:
- ✅ `[Feature] 添加章节选择界面`
- ✅ `[Bug] 故事播放界面布局错乱`
- ❌ ~~`新功能请求`~~
- ❌ ~~`故事播放有问题`~~

### 内容
Issue 的具体内容依据其类型而定. 你可以根据下面的实例 (模板)
来提出 Issue, 也可以按照自己的格式, 只要**足够清晰**即可.

#### 报告代码问题
示例 (模板):

```markdown
### 描述
简要描述 bug 的表现

### 复现步骤
1. 启动 '...'
2. 点击 '...'
3. 再点击 '...'
4. 出现错误

### 预期行为
清晰描述期望的结果

### 实际行为
描述实际发生了什么, 可附上错误信息

### 建议方案 (可选)
描述你期望的修复方式

### 截图/日志（可选）
粘贴截图或控制台/日志输出

### 环境信息 (可选)
- 操作系统: Windows 10
```

#### 功能请求
示例 (模板):

```markdown
### 背景
描述当前存在的痛点或使用场景

### 建议方案
描述你期望的功能或实现方式

### 备选方案 (可选)
列出考虑过的其他方案

### 附加信息 (可选)
截图、参考链接或原型设计
```

## ⛓ 分支管理

***如果你有权限直接管理仓库分支, 那么你应当遵守这些约定.***

*~~不是项目 Collaborator 就跳过这一节吧~~*

### 基本原则
**一个特性, 一个分支**. 原子性的特性分支有利于专注开发某一特性,
也有利于整理改动. 新功能是新功能, 修 bug 是修 bug, 请把它们拆分成多个分支.

**分支创建一般与 Issue 关联.** 见 [Issue # 什么时候提 Issue](#什么时候提-issue).

***严禁直接把代码推送到主分支!!!*** 直接推送主分支是**缺德行为**, 一定要写分支提 PR 给别人审查!

### 同一特性分支协作

若一个特性分支**只有你独自开发**, 则你可以**直接将代码推送到该分支**,
不需要额外创建个人分支, 实际上你可以把它当成你的个人分支.

若一个特性分支**需要多人协作**, 则每个人都应该在此分支的基础上, 创建出自己的分支,
然后分别在自己的分支上开发. 此时, 原有的特性分支称为**集成分支**.
每个人各自开分支, 也有利于减少冲突.

如果一个原本只有你独自开发的特性分支, 突然有**新成员加入开发**,
则你应当创建你的个人分支, 并将你的代码**转移至该个人分支上**.

**禁止不提 PR 直接推送至集成分支!** 你必须先推送到**自己的个人分支**.
然后**提出 PR**, 申请将个人分支合并到集成分支, 并接受审查.
要让别人检查你的代码, 才能放进集成分支里.

多人协作时, 为防止合并冲突, 推荐**多推送, 多合并**. 及时将已经写好的部分推送,
可以有效避免多人覆写同一份文件.

### 分支命名
*好的命名有利于管理不同分支.*

分支名称一律使用**全小写英文数字+横杠+正斜杠**的命名规则, 通常是形如 `category/name` 的名称,
这样符合标准命名规范, 也便于识别.

`category` 用于表明该分支属于哪些类型的开发. 以下是 `category` 的**通常参数**:

| 类型             | 使用场景                | 示例                                   |
|----------------|---------------------|--------------------------------------|
| **`feature`**  | 新功能开发               | `feature/potential-calculator`       |
| **`fix`**      | 修复 bug 或修改资源文件      | `fix/screen-metrics`                 |
| **`hotfix`**   | 更着急的 `fix`          | `hotfix/res-loc-audio-category`      |
| **`test`**     | 新增或修改单元测试           | `test/difficulty`                    |
| **`docs`**     | 新增或修改文档, 注释或 .md 文件 | `docs/contributing`                  |
| **`chore`**    | 修改其它配置              | `chore/upgrade-jackson`              |
| **`refactor`** | 代码逻辑重构, 但无业务影响      | `refactor/remove-deprecated-classes` |
| **`style`**    | 代码规范修改, 但没有任何逻辑变动   | `style/optimize-imports`             |
| **`perf`**     | 性能优化                | `perf/story-player-animation`        |
| **`archive`**  | 毙掉了, 但是想保留作纪念       | `archive/master`                     |

`name` 用于说明该分支主要内容, **不宜过长**, 可以适当缩写.

若该分支有对应的 Issue (大部分情况都有), 则向分支添加 `xxx-` 的 Issue 编号前缀,
以关联 Issue, 如 `fix/12-replace-image-placeholders`. 这样在 PR 通过后,
可以快速定位并关闭对应的 Issue.

对于个人分支, 应在原有集成分支名称的基础上**添加后缀** `-xxx`, 用于标识这是自己开发的分支,
如 `refactor/13-resource-loader-rikka`.

### 合并分支

通过提交 PR 来合并个人分支进集成分支, 或者合并集成分支进主分支.

有关规则见 [提交 PR](#-提交-pr).

### 生命周期
主分支 `main` **永久存在**, 遗留分支 `archive/master` **不再使用但仍保留**.

对于某个集成分支, 若其已经通过 PR 合并到主分支, 则应**及时清理**. 对于后续与之相关的一切变动,
应当**另外创建分支**.

对于集成分支附属的个人分支, 通常**跟随集成分支的生命周期**.

## 📰 Git 提交规范

### 基本原则
一次提交应当是**原子性的, 独立的, 能运行的最小单元**.

互不相干的改动, 应当拆成多个提交, 不要直接提交几百行复杂改动;
报错的半成品也不要端上来, 最起码注释掉报错的地方.

### 提交消息
按照流行的 Git 提交规范, 提交消息一般满足如下格式:

```text
[提交类型]: [具体消息]
```

以下是可用的 `[提交类型]`:

| 类型             | 使用场景                | 示例                               |
|----------------|---------------------|----------------------------------|
| **`feat`**     | 新功能                 | `feat: 完整故事播放功能实现`               |
| **`fix`**      | 修复 bug 或修改资源文件      | `fix: 屏幕长宽机制导致的布局错误`             |
| **`test`**     | 新增或修改单元测试           | `test: 添加 Difficulty 的单元测试`      |
| **`docs`**     | 新增或修改文档, 注释或 .md 文件 | `docs: 修改代码规范关于空行的要求`            |
| **`chore`**    | 修改其它配置              | `chore: 更新 Jackson 版本为 2.21.3`   |
| **`refactor`** | 代码逻辑重构, 但无业务影响      | `refactor: 提取故事节点重复使用的逻辑为公共静态方法` |
| **`style`**    | 代码规范修改, 但没有任何逻辑变动   | `style: 移除多余 import 语句`          |
| **`perf`**     | 性能优化                | `perf: 提升故事播放器的动画性能`             |

`[具体消息]` 不宜过长, 但要**说明这次提交具体干了什么**, 不要写过于笼统的消息:

❌ 错误写法:
- ~~`feat: Story 类`~~
- ~~`fix: 资源路径 bug`~~
- ~~`refactor: 重写故事显示逻辑`~~
- ~~`perf: 优化动画`~~

✅ 推荐写法:
- `feat: 表示故事的 Story 数据类`
- `fix: ResourceLocation.audio() 返回的实例中 category 不正确`
- `refactor: 将故事显示解耦为 View 层与 ViewModel 层`
- `perf: 提升故事播放动画的性能`

有些类型已经包含了行为动词, 此时在提交消息中, **推荐删除**冗余文本:
- `feat`: ~~`添加...的功能`~~
- `fix`: ~~`修复...的问题`~~
- `refactor`: ~~`重构...`~~
- `perf`: ~~`优化...`~~

省去动词, 或换用更具体的行为动词, 是更好的写法, 正如上面**推荐写法**的例子.

对于开头的动词, 按照流行规范, 推荐使用**祈使语气**, 而不用过去式:
- ✅ `补充 Difficulty 的单元测试`
- ❌ ~~`补充了 Difficulty 的单元测试`~~.

## 🔗 提交 PR

### 基本原则
提交 PR 前, 请确保你要合并的功能都**已经写完**, 不要把半成品扔进去.

任何 PR 都必须指定**至少一位审查者**, 并等待审查通过才能通过合并请求.
*(不要自己认为自己是审查者!!!)*

### 标题
类似 Git 提交消息规范, 把**最主要的改动**写出来, 具体内容放在描述内.
见 [Git 提交规范 # 提交消息](#提交消息).

### 描述
一份 PR 的描述要包含其**所有改动**, 清晰描述你具体**添加了什么**, **改了什么**,
**修复了什么**. 推荐使用序号 (`1.`, `2.`, `3.`...) 来逐个说明改动.

特性分支对应的 Issue 通常已经说明了实现该特性的背景和原因, 因此你**没有必要重复说明**.

### 跟进工作
**提交了 PR 不等于你结束工作了.** 你需要接受审查者的审查, 并根据其提出的建议进行**改动**,
然后推送. 直到审查者**同意 (Approved) 通过该 PR**, 你才能结束该分支的工作.

### 合并
***如果你有权限通过合并请求, 那么你应当遵守这些约定.***

只有在确保**审查没有问题**时才能通过合并请求. 你需要检查:
- 改动能否通过编译
- 存不存在 bug
- 提交符不符合规范

若没有问题, 则可以通过合并请求.

合并时, **选择 `Squash and merge`**, 然后指定提交消息与附加消息, 保证提交历史线性, 干净.

合并的提交消息与 PR 标题**保持一致** (或是略微修改以符合规范), 附加消息将所有改动附上,
并**关闭**该分支对应的 Issue (若有). 随后, 若该分支确认不再使用, 则应当**及时清理**.
